== GIT ==

git --help -> Mostra os comandos git.

ls -> Lista as pastas do diretório que estamos.

ls -a -> Lista todas as pastas do diretório, inclusive as pastas e arquivos ocultos.

cd ___ -> Para alterar o diretório. Ex: cd Desktop/ (para entrar na pasta 'Desktop' do diretório que estamos)

cd .. -> Retorna para a pasta acima da que nos encontramos.

pwd -> Mostra o caminho absoluta de onde estamos. O caminho que nos encontramos no momento.

*mkdir ___ -> Cria um diretório (uma pasta) dentro do diretório atual. mkdir = make a directory.

cat ___ -> Mostra o que está dentro do arquivo que colocamos para ler.

git init . -> Transforma a pasta em uma pasta passível de receber um repositório Git. Gera uma pasta oculta chamada '.git/', que é o que torna essa pasta especial. Para desfazer isso, é necessário apagar a pasta '.git/' do diretório, utilizando o comando 'rm -rf .git/'.

rm -rf ___ -> Exclui a pasta descrita e tudo nela. MUITO CUIDADO AO UTILIZAR ESTE COMANDO!!!!!

*git status -> Me traz os commits realizados e arquivos que não possuem versionamento (que não foram trakeados para versionar).

*git add ___ -> Adiciona o arquivo na lista dos arquivos a serem commitados (para ser versionado).

*git add . -> Adiciona todos os arquivos da pasta atual para ser commitado.

*git reset <nome do arquivo> -> Retira o arquivo do commit.

*git commit -m "Título que eu quiser dar para o checkpoint" -> Para criar o commit de fato, já o nomeando. O nome é para identificar a alteração a se fazer ou o ponto ao qual o código estava antes do commit ser realizado.

git config --global user.email "email@email.com" -> Para adicionar a identificação de email de quem esta commitando. Basicamente a assinatura de quem está alterando o código.

git config --global user.name "nome usuário" -> Para adicionar a identificação pelo nome do usuário. Usa-se os dois comando para adicionar a identificação, primeiro um e depois o outro.

*git config --global user.name (OU) user.email -> Para mostrar qual a 'assinatura' ativa no momento.

nano ___ -> É um editor de texto dentro do Bash, para alterar o arquivo que estamos commitando.

*git log -> Mostra o histórico de todos os commits feitos.

*git diff <nome do arquivo> -> Mostra as alterações do estado atual do arquivo para o commit anterior. IMPORTANTE DEMAIS.

*git checkout -b <nome da 'linha do tempo'/branch> -> Para criar uma nova branch (uma nova ramificação), isolando os arquivos para mudanças que não alterem a branch principal. Como se criasse uma cópia dos arquivos para modifica-los.

*git branch -> Mostra todas as branch's do nosso projeto/repositório Git.

*git checkout "nome de alguma branch já existente" -> Passamos a estar na branch selecionada.

*git merge <nome da branch a ser trazida> -> 'Mistura' a branch que nos encontramos com a descrita no comando, como se trouxesse o 'progresso' da selecionada para a atual. Traz as ALTERAÇÕES da outra branch, e não os arquivos. Caso tenham sido criado arquivos novos, o arquivo vem.

*git branch -D <nome da branch> -> Apaga a branch descrita. Feito após o 'merge' para fazer uma limpeza em nosso diretório.

git clone <url do repositório ou fork> -> Cria um repositório igual ao que está na nuvem, porém na máquina que estamos trabalhando. Assim podemos trabalhar em nosso computador de forma local.

## GITHUB

git remote add origin <url do repositório do github> -> Cria o link/ligação com o repositório do GitHub.

git push origin main -> Puxa todos os arquivos e commits do repositório local (main) para o repositório em nuvem (origin, que no caso é no GitHub). Toda vez que fizermos uma alteração em nosso repositório local, ela não será enviada automaticamente para o GitHub. Precisamos realizar o comando 'git push origin main' todas as vezes que quisermos atualizar nosso repositório em nuvem!

git pull origin main -> Puxa todos os arquivos e commits do repositório em nuvem (origin) para o repositório local (main). Quando fazemos alterações na nuvem, precisamos fazer isso para atualizar o repositório em nosso computador.

##

Versionamento: é o processo de registrar e gerenciar alterações em arquivos, códigos ou documentos ao longo do tempo, atribuindo nomes ou números únicos (versões) a cada modificação. Ele permite rastrear o histórico, restaurar estados anteriores e colaborar de forma eficiente, sendo essencial para desenvolvimento de software.

* Pull Request: é a sinalização para GitHub, de que pretendemos realizar um merge. Em geral, é a solicitação de merge que ficará disponível para que possam aprová-la, antes do merge ser realizado de fato. "Solicitação para puxar".

(main): Se refere a versão principal de nosso arquivo. 'História principal'.

## FLUXO DE TRABALHO

4. git add <nome do arquivo>
5. git status : Para verificar se são os arquivos que queremos commitar mesmo.
6. git commit -m "mensagem do commit"
7. volta para o passo 3
